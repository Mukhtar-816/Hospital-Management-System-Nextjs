## 1. Project Overview

The **Hospital Management System (HMS)** is a full-stack web application designed to digitise and manage the core operational workflows of a hospital environment. It brings together four distinct user roles — **Admin, Doctor, Receptionist, and Patient** — under one unified platform, each with precisely scoped access rights, a dedicated dashboard, and a defined set of permitted actions.

The system is underpinned by a relational PostgreSQL database that enforces referential integrity, domain constraints, and a carefully designed Role-Based Access Control (RBAC) model.

### 1.1 Objectives

- Provide a centralised, role-aware digital platform for managing hospital operations.
- Enforce data integrity through primary keys, foreign keys, unique constraints, and check constraints.
- Implement a full appointment lifecycle with deterministic status transitions.
- Record clinical outcomes (interactions, diagnoses, prescriptions) atomically using database transactions.
- Deliver real-time, role-specific dashboard statistics using aggregation queries.
- Protect sensitive patient data through JWT-based authentication and RBAC permission checks.

### 1.2 Technology Stack

The application is built as a monorepo. The frontend and API layer are part of the same Next.js 15 application. The DBMS layer is entirely PostgreSQL.

| Layer | Technology | Role in System |
|---|---|---|
| Frontend UI | Next.js 15, React, TypeScript | Server-side rendered pages, client components |
| Styling | Tailwind CSS | Utility-first responsive design |
| API Layer | Next.js Route Handlers (REST) | Exposes HTTP endpoints consumed by the UI |
| Database | PostgreSQL 15+ | Relational DBMS — sole source of truth |
| DB Driver | node-postgres (pg) | Connection pooling & parameterised queries |
| Auth | JWT (jsonwebtoken) + bcryptjs | Session tokens & password hashing |
| Middleware | Next.js Middleware (jose) | Route protection & role-based redirects |
| Extensions | pgcrypto (PostgreSQL) | UUID generation with `gen_random_uuid()` |

## 2. Database Schema — Table-by-Table Analysis

The database consists of **thirteen tables** divided into two logical groups: the **RBAC infrastructure tables** and the **domain (hospital) tables**. Every primary key is a UUID generated by PostgreSQL's `pgcrypto` extension using `gen_random_uuid()`, ensuring globally unique, non-sequential identifiers.

### 2.1 RBAC Infrastructure Tables

RBAC is implemented through five tables encoding a many-to-many relationship between users, roles, and permissions — a textbook implementation of the **NIST RBAC Standard Level 1** model.

#### 2.1.1 Users Table

```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

CREATE TABLE Users (
  userId       UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  userEmail    VARCHAR(100) UNIQUE NOT NULL,
  userPassword VARCHAR(255) NOT NULL,
  createdAt    TIMESTAMP    DEFAULT NOW(),
  updatedAt    TIMESTAMP    DEFAULT NOW()
);
```

**Purpose:** Central identity store. Every actor in the system must have exactly one row here before any domain-level profile is created.

| Column | Type | Constraint | Notes |
|---|---|---|---|
| userId | UUID | PK | Auto-generated by pgcrypto. Non-sequential to prevent enumeration attacks. |
| userEmail | VARCHAR(100) | UNIQUE NOT NULL | Login identifier. Uniqueness prevents duplicate accounts. |
| userPassword | VARCHAR(255) | NOT NULL | Stores bcrypt hash (60 chars). VARCHAR(255) accommodates future algorithm changes. |
| createdAt | TIMESTAMP | DEFAULT NOW() | Audit trail — records exact moment of account creation. |
| updatedAt | TIMESTAMP | DEFAULT NOW() | Automatically updated by `trg_users_updated` trigger on every UPDATE. |

#### 2.1.2 Role Table

```sql
CREATE TABLE Role (
  roleId  UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  name    VARCHAR(50) UNIQUE NOT NULL
);

-- Seeded values:
INSERT INTO Role (name) VALUES ('admin'), ('doctor'), ('receptionist'), ('patient');
```

#### 2.1.3 Permission Table

```sql
CREATE TABLE Permission (
  permissionId  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  name          VARCHAR(100) UNIQUE NOT NULL
);

-- Seeded permissions:
-- USER:         create_user, read_user, update_user
-- PATIENT:      create_patient, read_patient, update_patient
-- DOCTOR:       create_doctor, read_doctor, update_doctor
-- RECEPTIONIST: create_receptionist
-- COMMON:       read_own_data
```

#### 2.1.4 UserRole Table (Junction)

```sql
CREATE TABLE UserRole (
  id      UUID  PRIMARY KEY DEFAULT gen_random_uuid(),
  userId  UUID  NOT NULL,
  roleId  UUID  NOT NULL
);

ALTER TABLE UserRole
  ADD CONSTRAINT fk_userrole_user
    FOREIGN KEY (userId) REFERENCES Users(userId) ON DELETE CASCADE,
  ADD CONSTRAINT fk_userrole_role
    FOREIGN KEY (roleId) REFERENCES Role(roleId)  ON DELETE CASCADE,
  ADD CONSTRAINT unique_user_role UNIQUE (userId, roleId);
```

> **Design note:** `assignRole()` first DELETEs all existing UserRole rows for a `userId` before inserting a new one, enforcing a **single-role-per-user** policy at the application level.

#### 2.1.5 RolePermission Table (Junction)

```sql
CREATE TABLE RolePermission (
  id            UUID  PRIMARY KEY DEFAULT gen_random_uuid(),
  roleId        UUID  NOT NULL,
  permissionId  UUID  NOT NULL
);

ALTER TABLE RolePermission
  ADD CONSTRAINT fk_roleperm_role
    FOREIGN KEY (roleId)       REFERENCES Role(roleId)       ON DELETE CASCADE,
  ADD CONSTRAINT fk_roleperm_perm
    FOREIGN KEY (permissionId) REFERENCES Permission(permissionId) ON DELETE CASCADE,
  ADD CONSTRAINT unique_role_permission UNIQUE (roleId, permissionId);
```

### 2.2 Domain Tables

#### 2.2.1 Patient Table

```sql
CREATE TABLE Patient (
  patientId     UUID      PRIMARY KEY DEFAULT gen_random_uuid(),
  userId        UUID      UNIQUE,
  patientNumber BIGSERIAL UNIQUE,
  fullname      VARCHAR(100) NOT NULL,
  address       TEXT,
  gender        VARCHAR(20),
  createdAt     TIMESTAMP DEFAULT NOW(),
  updatedAt     TIMESTAMP DEFAULT NOW()
);

ALTER TABLE Patient
  ADD CONSTRAINT fk_patient_user
    FOREIGN KEY (userId) REFERENCES Users(userId) ON DELETE SET NULL;
```

> `ON DELETE SET NULL` — if the user account is deleted, the patient record is **preserved** with `userId = NULL`, retaining medical history. Critical data-integrity decision in a medical context.

#### 2.2.2 Doctor Table

```sql
CREATE TABLE Doctor (
  doctorId       UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  userId         UUID         UNIQUE NOT NULL,
  fullname       VARCHAR(100) NOT NULL,
  specialization VARCHAR(100),
  status         VARCHAR(50),
  createdAt      TIMESTAMP    DEFAULT NOW(),
  updatedAt      TIMESTAMP    DEFAULT NOW()
);

ALTER TABLE Doctor
  ADD CONSTRAINT fk_doctor_user
    FOREIGN KEY (userId) REFERENCES Users(userId) ON DELETE CASCADE;
```

> `ON DELETE CASCADE` here — a doctor's availability profile is meaningless without the account, unlike patients.

#### 2.2.3 Receptionist Table

```sql
CREATE TABLE Receptionist (
  receptionistId  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  userId          UUID         UNIQUE NOT NULL,
  fullname        VARCHAR(100) NOT NULL,
  createdAt       TIMESTAMP    DEFAULT NOW(),
  updatedAt       TIMESTAMP    DEFAULT NOW()
);

ALTER TABLE Receptionist
  ADD CONSTRAINT fk_receptionist_user
    FOREIGN KEY (userId) REFERENCES Users(userId) ON DELETE CASCADE;
```

#### 2.2.4 Appointment Table

```sql
CREATE TABLE Appointment (
  appointmentId   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  patientId       UUID        NOT NULL,
  doctorId        UUID        NOT NULL,
  receptionistId  UUID,
  requestId       UUID,
  starttime       TIMESTAMP   NOT NULL,
  endtime         TIMESTAMP,
  type            VARCHAR(50) DEFAULT 'walk-in',
  status          VARCHAR(50) DEFAULT 'scheduled'
);
```

| Column | Description |
|---|---|
| status | Lifecycle: `scheduled` → `checked_in` → `in_progress` → `completed` / `no_show` |
| endtime | Set to `starttime + 30 min` at insert; updated to `NOW()` on completion |
| receptionistId | Nullable — walk-ins can be created without a receptionist |

#### 2.2.5 Interaction, Diagnosis & Prescription Tables

```sql
-- Clinical record root — one per completed appointment
CREATE TABLE Interaction (
  interactionId  UUID      PRIMARY KEY DEFAULT gen_random_uuid(),
  appointmentId  UUID      NOT NULL UNIQUE,  -- enforces 1:1 with Appointment
  notes          TEXT,
  createdAt      TIMESTAMP DEFAULT NOW()
);

-- One-to-many: multiple diagnoses per interaction
CREATE TABLE Diagnosis (
  diagnosisId    UUID  PRIMARY KEY DEFAULT gen_random_uuid(),
  interactionId  UUID  NOT NULL,
  description    TEXT  NOT NULL
);

-- One-to-many: multiple prescriptions per interaction
CREATE TABLE Prescription (
  prescriptionId  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  interactionId   UUID         NOT NULL,
  medicine        VARCHAR(200) NOT NULL,
  frequency       VARCHAR(100),
  duration        VARCHAR(100),
  instructions    TEXT
);
```

## 3. Entity-Relationship Diagram & Relational Model

### 3.1 Complete Relationship Catalogue

| Entity A | Cardinality | Entity B | FK Column | On Delete Behaviour |
|---|---|---|---|---|
| Users | 1 : 0..1 | Patient | Patient.userId | SET NULL — preserves medical history |
| Users | 1 : 0..1 | Doctor | Doctor.userId | CASCADE — profile meaningless without account |
| Users | 1 : 0..1 | Receptionist | Receptionist.userId | CASCADE |
| Users | 1 : N | UserRole | UserRole.userId | CASCADE |
| Role | 1 : N | UserRole | UserRole.roleId | CASCADE |
| Role | 1 : N | RolePermission | RolePermission.roleId | CASCADE |
| Permission | 1 : N | RolePermission | RolePermission.permissionId | CASCADE |
| Patient | 1 : N | Appointment | Appointment.patientId | RESTRICT |
| Doctor | 1 : N | Appointment | Appointment.doctorId | RESTRICT |
| Receptionist | 1 : N | Appointment | Appointment.receptionistId | NULLABLE |
| Appointment | 1 : 0..1 | Interaction | Interaction.appointmentId | CASCADE |
| Interaction | 1 : N | Diagnosis | Diagnosis.interactionId | CASCADE |
| Interaction | 1 : N | Prescription | Prescription.interactionId | CASCADE |

### 3.2 Normalisation Analysis

The schema adheres to **Third Normal Form (3NF)**:

- **1NF:** Every column contains atomic values. Diagnoses are individual rows, not comma-separated strings.
- **2NF:** All non-key attributes are fully functionally dependent on the entire primary key.
- **3NF:** No transitive dependencies. Patient demographics in `Patient`, clinical data in `Interaction`/`Diagnosis`/`Prescription`, never duplicated in `Appointment`.

## 4. SQL Queries — In-Depth Analysis

### 4.1 Authentication & User Management

#### Login — User Lookup by Email
```sql
SELECT * FROM users WHERE useremail = $1
```
B-Tree index on `userEmail` (from UNIQUE constraint) makes this O(log n). Parameterised `$1` prevents SQL injection.

#### Get User Roles (Post-Login Token Building)
```sql
SELECT r.name
FROM role r
JOIN userrole ur ON r.roleid = ur.roleid
WHERE ur.userid = $1
```

#### Assign Role (Admin Action)
```sql
-- Step 1: Resolve role name to UUID
SELECT roleid FROM role WHERE name = $1

-- Step 2: Remove any existing role assignments (enforces single-role invariant)
DELETE FROM userrole WHERE userid = $1

-- Step 3: Insert new role assignment
INSERT INTO userrole (userid, roleid) VALUES ($1, $2)
```

---

### 4.2 Admin Dashboard — Multi-Table Aggregation

```sql
SELECT
  u.userid,
  u.useremail,
  r.name                                              AS role,
  COALESCE(d.fullname, rec.fullname, p.fullname)     AS fullname,
  p.address, p.gender, p.patientid,
  d.specialization, d.doctorid,
  rec.receptionistid
FROM users u
JOIN userrole ur    ON u.userid  = ur.userid
JOIN role r         ON ur.roleid = r.roleid
LEFT JOIN doctor      d   ON d.userid   = u.userid
LEFT JOIN receptionist rec ON rec.userid = u.userid
LEFT JOIN patient     p   ON p.userid   = u.userid
ORDER BY u.createdat DESC
```

**Key techniques:**
- `INNER JOIN` to `userrole`/`role` — only users with assigned roles appear.
- `LEFT JOIN` to domain tables — a given user only has a row in **one** of the three tables.
- `COALESCE(d.fullname, rec.fullname, p.fullname)` — returns the first non-NULL name regardless of role.

### 4.3 Appointment Queries

#### Conflict Check Before Booking
```sql
SELECT 1
FROM appointment
WHERE doctorid = $1
  AND starttime = $2
  AND status != 'cancelled'
```
`SELECT 1` checks existence only. `status != 'cancelled'` ensures cancelled slots are re-bookable.

#### Create Appointment
```sql
INSERT INTO appointment (
  patientid, doctorid, receptionistid, requestid,
  starttime, endtime, type, status
) VALUES ($1, $2, $3, $4, $5, $6, $7, 'scheduled')
RETURNING *
```
`status` hardcoded to `'scheduled'` — no appointment can be born in any other state. `RETURNING *` avoids a follow-up SELECT.

#### Doctor Availability Search
```sql
SELECT
  d.doctorid, d.fullname, d.specialization,
  CASE
    WHEN a.appointmentid IS NOT NULL THEN TRUE
    ELSE FALSE
  END AS is_busy
FROM doctor d
LEFT JOIN Appointment a
  ON  d.doctorid  = a.doctorid
  AND a.starttime = $1::TIMESTAMP
  AND a.status NOT IN ('cancelled', 'rejected')
WHERE 1=1
-- [AND d.specialization = $2]  -- appended dynamically if filter provided
```

`LEFT JOIN` ensures all doctors appear. `CASE` produces a boolean `is_busy` flag. `WHERE 1=1` is a safe dynamic SQL pattern for optional filters.

#### Appointment Status Transitions
```sql
-- CHECK-IN (Receptionist: scheduled → checked_in)
UPDATE appointment SET status = 'checked_in'  WHERE appointmentid = $1 RETURNING *

-- START (Doctor: checked_in → in_progress, actualise starttime)
UPDATE appointment SET status = 'in_progress', starttime = NOW() WHERE appointmentid = $1 RETURNING *

-- COMPLETE (Doctor: in_progress → completed, record real endtime)
UPDATE appointment SET status = 'completed',   endtime   = NOW() WHERE appointmentid = $1 RETURNING *

-- NO-SHOW (Receptionist: scheduled → no_show)
UPDATE appointment SET status = 'no_show'      WHERE appointmentid = $1 RETURNING *
```

### 4.4 Clinical Data — ACID Transaction

```sql
BEGIN;

-- Step 1: Create Interaction header
INSERT INTO interaction (appointmentid, notes)
VALUES ($1, $2)
RETURNING interactionid;

-- Step 2: Batch insert all diagnoses (N rows in one query)
INSERT INTO diagnosis (interactionid, description)
VALUES ($1, $2), ($1, $3), ($1, $4)  -- dynamic based on count

-- Step 3: Batch insert all prescriptions
INSERT INTO prescription (interactionid, medicine, frequency, duration, instructions)
VALUES ($1, $2, $3, $4, $5), ($1, $6, $7, $8, $9)  -- dynamic

-- Step 4: Mark appointment completed
UPDATE appointment SET status = 'completed' WHERE appointmentid = $1;

COMMIT;
-- On any error: ROLLBACK automatically reverts all four steps
```

This is the **only** query group using an explicit transaction. The four operations form an atomic clinical unit — partial failure is impossible.

> **Batch INSERT efficiency:** N diagnoses/prescriptions are inserted in a single multi-row `VALUES` clause, reducing round-trips from N to 1.

### 4.5 Clinical Data Retrieval — `json_agg`

```sql
SELECT
  i.interactionid,
  i.notes,
  json_agg(DISTINCT jsonb_build_object('description', d.description))
    FILTER (WHERE d.description IS NOT NULL) AS diagnoses,
  json_agg(DISTINCT jsonb_build_object(
    'medicine',     p.medicine,
    'frequency',    p.frequency,
    'duration',     p.duration,
    'instructions', p.instructions
  )) FILTER (WHERE p.medicine IS NOT NULL)   AS prescriptions
FROM interaction i
LEFT JOIN diagnosis    d ON i.interactionid = d.interactionid
LEFT JOIN prescription p ON i.interactionid = p.interactionid
WHERE i.appointmentid = $1
GROUP BY i.interactionid, i.notes
```

- `json_agg(DISTINCT ...)` — aggregates joined rows into a JSON array; `DISTINCT` prevents duplicates from the cross-join effect.
- `FILTER (WHERE ... IS NOT NULL)` — returns `NULL` instead of `[null]` when no child rows exist.

### 4.6 Dashboard Queries

#### Doctor Dashboard Stats (Single Scan)
```sql
SELECT
  COUNT(*) FILTER (WHERE starttime::date = CURRENT_DATE) AS today_appointments,
  COUNT(*) FILTER (WHERE status = 'completed')           AS patients_seen,
  COUNT(*) FILTER (WHERE status = 'in_progress')         AS pending_records
FROM appointment
WHERE doctorid = (SELECT doctorid FROM doctor WHERE userid = $1)
```
Three metrics in **one table scan** using conditional `COUNT...FILTER`.

#### Patient Activity Feed — UNION ALL
```sql
SELECT * FROM (
  SELECT
    'Appointment'             AS type,
    a.starttime               AS date,
    a.status,
    'Appointment with ' || d.fullname AS title,
    a.appointmentid::text     AS id
  FROM appointment a
  JOIN doctor d ON a.doctorid = d.doctorid
  WHERE a.patientid = (SELECT patientid FROM patient WHERE userid = $1)

  UNION ALL

  SELECT
    'Record'                  AS type,
    i.createdat               AS date,
    'info'                    AS status,
    'Medical record updated'  AS title,
    i.interactionid::text     AS id
  FROM interaction i
  JOIN appointment a ON i.appointmentid = a.appointmentid
  WHERE a.patientid = (SELECT patientid FROM patient WHERE userid = $1)
) AS combined_activity
ORDER BY date DESC
LIMIT 5
```
`UNION ALL` (not `UNION`) avoids unnecessary deduplication overhead — the two branches can never produce identical rows.

## 5. Database Triggers

```sql
-- Shared trigger function
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updatedAt = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Applied to four tables:
CREATE TRIGGER trg_users_updated       BEFORE UPDATE ON Users        FOR EACH ROW EXECUTE FUNCTION set_updated_at();
CREATE TRIGGER trg_patient_updated     BEFORE UPDATE ON Patient      FOR EACH ROW EXECUTE FUNCTION set_updated_at();
CREATE TRIGGER trg_doctor_updated      BEFORE UPDATE ON Doctor       FOR EACH ROW EXECUTE FUNCTION set_updated_at();
CREATE TRIGGER trg_receptionist_updated BEFORE UPDATE ON Receptionist FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

- **`BEFORE UPDATE`** — fires before the statement commits, modifying `NEW` row before it hits disk.
- **`FOR EACH ROW`** — fires per modified row; correct for both single and batch updates.
- **Database-enforced** — `updatedAt` is always current, even if a developer forgets to include it in the `SET` clause.
- **Scope:** Only on user-editable profile tables. `Appointment`, `Interaction`, `Diagnosis`, `Prescription` are append-only by design.

## 6. Data Integrity Constraints

### 6.1 Database-Level Constraints

| Table | Column | Constraint | Effect |
|---|---|---|---|
| Users | userId | PRIMARY KEY | Row uniqueness; auto-UUID |
| Users | userEmail | UNIQUE NOT NULL | No duplicate accounts |
| Users | userPassword | NOT NULL | No account without password hash |
| Role | name | UNIQUE NOT NULL | No duplicate role names |
| Permission | name | UNIQUE NOT NULL | No duplicate permission strings |
| UserRole | (userId, roleId) | UNIQUE | No duplicate role assignments |
| RolePermission | (roleId, permId) | UNIQUE | No duplicate permission assignments |
| Patient | userId | UNIQUE | One patient profile per user |
| Patient | patientNumber | UNIQUE BIGSERIAL | Sequential human-readable patient ID |
| Doctor | userId | UNIQUE NOT NULL | One doctor profile per user |
| Receptionist | userId | UNIQUE NOT NULL | One receptionist profile per user |

### 6.2 Application-Level Constraints

- **Email pre-check:** `findByEmail()` before insert provides user-friendly error ahead of DB constraint violation.
- **Appointment conflict check:** `SELECT` before `INSERT` — no DB `UNIQUE(doctorid, starttime)` constraint exists; this is the sole enforcement mechanism.
- **Status transition validation:** Predecessor status fetched and validated before every `UPDATE`.
- **Permission enforcement:** `requirePermission()` checks live DB data on every request — not cached JWT claims.
- **Ownership checks:** `GET /api/appointments/[id]` verifies requesting user is the doctor, patient, or a receptionist.

## 7. Authentication & Security Architecture

### 7.1 Registration Flow

1. User submits `email + password + fullname` via `POST /api/auth/register`.
2. `findByEmail()` — rejects if account already exists.
3. `bcrypt.genSalt(10)` + `bcrypt.hash()` — cost factor 10 (2¹⁰ = 1024 iterations).
4. `createUser()` — inserts into `Users`.
5. `assignRole()` — inserts `'patient'` role into `UserRole`.
6. `createPatient()` — creates `Patient` profile row.
7. All self-registrations get the `'patient'` role. Only Admin can elevate.

### 7.2 Login Flow

1. `findByEmail()` fetches user record.
2. `bcrypt.compare()` validates password against stored hash.
3. `getUserRoles()` fetches role from RBAC tables.
4. `jwt.sign({ userid, role }, secret, { expiresIn: '7d' })`.
5. Token set as **`httpOnly` cookie** — prevents XSS token theft.

### 7.3 Request Authentication Flow

1. `getUser(req)` reads the `'token'` cookie.
2. `jwt.verify()` validates signature and expiry.
3. `getUserRoleAndPermissions()` fetches **live DB data** — ensures role changes take immediate effect.
4. `requirePermission()` checks permission list; throws `403 Forbidden` if not found.

### 7.4 Middleware Route Protection

Next.js middleware intercepts every page navigation request:
- Redirects authenticated users away from `/login`, `/register` to their role dashboard.
- Redirects unauthenticated users to `/login`.
- Prevents cross-role URL access (e.g., a doctor accessing `/patient/...` is redirected to `/doctor/dashboard`).

## 8. Appointment Lifecycle — State Machine

```
                        ┌─────────────────────────────────────┐
                        │         APPOINTMENT STATES           │
                        └─────────────────────────────────────┘

   [Receptionist]            [Receptionist]           [Doctor]            [Doctor]
  POST /appointments    PATCH .../checkin          PATCH .../start    POST .../interaction
        │                      │                       │                    │
        ▼                      ▼                       ▼                    ▼
  ┌──────────┐          ┌────────────┐          ┌────────────┐       ┌──────────┐
  │scheduled │─────────▶│ checked_in │─────────▶│ in_progress│──────▶│completed │
  └──────────┘          └────────────┘          └────────────┘       └──────────┘
        │
        │ [Receptionist]
        │ PATCH .../no-show
        ▼
  ┌──────────┐
  │ no_show  │  (terminal)
  └──────────┘
```

| From | Action | To | Actor | Endpoint |
|---|---|---|---|---|
| _(none)_ | Create | `scheduled` | Receptionist | `POST /api/appointments` |
| `scheduled` | Check-In | `checked_in` | Receptionist | `PATCH .../checkin` |
| `scheduled` | No-Show | `no_show` | Receptionist | `PATCH .../no-show` |
| `checked_in` | Start | `in_progress` | Doctor | `PATCH .../start` |
| `in_progress` | Save Clinical Data | `completed` | Doctor | `POST .../interaction` |
| `in_progress` | Complete | `completed` | Doctor | `PATCH .../complete` |

**Key design decisions:**
- A doctor cannot start until receptionist checks in — physical presence confirmed before clinical time starts.
- `no_show` is only reachable from `scheduled` — a checked-in patient cannot be marked no-show.
- Every transition validates the predecessor state; illegal transitions return `HTTP 400`.

## 9. Application Flow & Role-Specific Features

### 9.1 Admin
- Views all users via the `COALESCE LEFT JOIN` query across all domain tables.
- Creates, deletes, and promotes users via `/api/admin/users`.
- Has all 11 permissions (via `CROSS JOIN` seeding).

### 9.2 Receptionist
- Views all appointments (INNER JOIN on appointment + patient + doctor).
- Creates appointments: patient search → doctor availability check → conflict check → INSERT.
- Manages check-in and no-show status transitions.

### 9.3 Doctor
- Personalised dashboard with conditional `COUNT FILTER` stats.
- Starts consultations (`checked_in → in_progress`).

## 10. System Restrictions & How They Are Enforced

| Restriction | Layer | Mechanism |
|---|---|---|
| No duplicate email addresses | DB + App | `UNIQUE(userEmail)`; `findByEmail()` pre-check |
| One role per user | App | `assignRole()` DELETEs before INSERT |
| One profile per user | DB | `UNIQUE(userId)` on Patient, Doctor, Receptionist |
| No double-booking of doctors | App | Conflict `SELECT` before `INSERT` |
| Sequential appointment state transitions | App | Status pre-validation before every `UPDATE` |
| Clinical data is all-or-nothing | DB | `BEGIN/COMMIT` transaction over 4 operations |
| Patients cannot see other patients' data | App | Ownership check in `GET /api/appointments/[id]` |
| Only doctors can submit clinical data | App | `requirePermission("clinical.write")` |
| Only receptionists can check-in | App | `requirePermission("appointment.update")` |
| `updatedAt` always reflects last change | DB | `BEFORE UPDATE` trigger on 4 tables |
| Cascade deletions maintain referential integrity | DB | `ON DELETE CASCADE` on junction + domain tables |
| Patient records survive account deletion | DB | `ON DELETE SET NULL` on `Patient.userId` FK |
| Unauthenticated users blocked | App (Middleware) | JWT verification in `proxy.ts` |
| Cross-role page access blocked | App (Middleware) | Pathname prefix check → redirect to own dashboard |

## 11. Database Connection — Pooling Architecture

```typescript
// lib/db.ts
import { Pool } from 'pg';

const globalForPG = globalThis as unknown as { pool: Pool | undefined };

export const pool = globalForPG.pool ?? new Pool({
  connectionString: process.env.DATABASE_URL,
  ssl: process.env.DATABASE_URL?.includes("localhost")
    ? false
    : { rejectUnauthorized: true },
});

if (!globalForPG.pool) {
  globalForPG.pool = pool;
}
```

- **Singleton via `globalThis`:** Prevents connection exhaustion during Next.js hot-module reloading in development.
- **SSL configuration:** Disabled for `localhost`; enforced with `rejectUnauthorized: true` in production to prevent MITM attacks.
- **`DATABASE_URL`:** Compatible with all major cloud PostgreSQL providers (Supabase, Neon, Railway, AWS RDS).
